📌 ข้อสรุปสำหรับการตัดสินใจ

  • Go ชนะ 14 จาก 24 scenarios (58%) — ทดสอบหลัง reboot VM (cold start สะอาด) เป็น final result ที่น่าเชื่อถือที่สุด
  • Java ชนะ .txt/.csv (6 จาก 6 scenarios) — JVM ZLIB path เร็วกว่าสำหรับข้อมูลที่บีบอัดได้สูง
  • Go ชนะทุก binary file (12/12 scenarios) — .pdf, .xlsx, .gz, .dat Go เร็วกว่า 1.2–1.4× เพราะ AES hardware path ของ Go runtime มีประสิทธิภาพสูงกว่า
  • Curve25519 — Go ได้เปรียบมากที่สุด (4/7 scenarios, เร็วกว่า 1.2–1.2×) — Go ECC path เร็วกว่า Bouncy Castle อย่างชัดเจน
  • Concurrent load (8 clients พร้อมกัน) — Go เร็วกว่า 1.8× — Go scale ได้ดีกว่าเมื่อ load สูง Java throughput ตกมากกว่า
  • ⭐ ข้อแนะนำ: Go เหมาะสำหรับระบบใหม่ที่ต้องการ PGP performance สูงสุด — โดยเฉพาะถ้าใช้ Curve25519 หรือมี concurrent load สูง
    Java ทำได้ดีพอ สำหรับระบบ enterprise ที่ใช้ Java stack อยู่แล้ว และ workload ไม่ sensitive กับ ~1.4× performance gap

🔒 PGP คืออะไร ทำไมต้องสนใจ?

PGP (Pretty Good Privacy) คือมาตรฐานการเข้ารหัสไฟล์และข้อมูล ที่ใช้กันทั่วโลกมากกว่า 30 ปี ทำงานเหมือนใส่กุญแจล็อคไฟล์ ด้วย "กุญแจสาธารณะ" (public key) และเปิดได้เฉพาะคนที่มี "กุญแจส่วนตัว" (private key) เท่านั้น ถ้าไฟล์หลุดออกไปก็อ่านไม่ได้โดยไม่มีกุญแจ

⏱ ms คืออะไร ยิ่งน้อยดีมะ?

ms = มิลลิวินาที (1 วินาที = 1,000 ms) ใช่ — ยิ่งน้อย ยิ่งเร็ว ยิ่งดี
เพื่อช่วยจินตนาการ: การกระพริบตา 1 ครั้ง ≈ 150 ms ระบบที่ตอบสนอง <100 ms คนไม่รู้สึกว่า "ช้า" สำหรับ batch process ไฟล์หลายพัน — ทุก ms สะสมกันได้เป็นนาที

📊 p50 คืออะไร ต่างจาก average ยังไง?

p50 (median — ค่ากลาง): เรียงตัวเลขทุกค่าจากน้อยไปมาก แล้วเอาค่าตรงกลาง — ไม่ถูกดึงเบี้ยวโดยค่าสูงผิดปกติ (outlier)
Average: ถ้ามีค่า spike ครั้งเดียวสูงมาก ค่าเฉลี่ยพุ่งขึ้นแม้ส่วนใหญ่ปกติ p50 บอกว่า "ครึ่งหนึ่งของการทดสอบ เร็วกว่าตัวเลขนี้" — สะท้อนความเป็นจริงกว่า

🆚 Go และ Java ต่างกันยังไงในบริบทนี้?

Go: ภาษาสมัยใหม่จาก Google ซอฟต์แวร์ compile เป็น binary เดียวพร้อมรัน ไม่ต้องติดตั้ง runtime เริ่มต้นเร็ว ใช้หน่วยความจำน้อย
Java: ภาษาที่ใช้กันมากกว่า 30 ปี ต้องมี JVM (Java Virtual Machine) รันอยู่ก่อน มี JIT compiler ที่อุ่นเครื่องแล้วเร็วมาก แต่ช่วงแรกช้ากว่า ระบบ Enterprise ส่วนใหญ่ใช้ Java

🎯 ผลนี้เชื่อถือได้แค่ไหน?

เชื่อถือได้สูง เพราะมีกลไกป้องกันหลายชั้น:
✅ 721 tests ผ่านทั้งหมด ก่อนรัน benchmark
✅ สลับลำดับ Go/Java ทุกรอบ — ไม่มีฝ่ายได้เปรียบ
✅ ตรวจความถูกต้องของข้อมูลทุก operation
✅ วัดเฉพาะการเข้ารหัสจริง ไม่รวมการโหลดไฟล์
⚠️ ข้อจำกัด: ทดสอบบน VM ผล bare metal อาจต่างกัน 10–20% แต่ความต่าง Go vs Java น่าจะยังเหมือนเดิม

➡️ ต้องทำอะไรต่อ?

ขึ้นกับว่าระบบเป้าหมายเป็นอย่างไร:

POC นี้มีเป้าหมายเดียวที่ชัดเจน: เปรียบเทียบประสิทธิภาพการเข้ารหัส PGP ระหว่างภาษา Go และ Java ภายใต้เงื่อนไขที่ยุติธรรมและทำซ้ำได้ เพื่อช่วยตัดสินใจเลือกภาษาสำหรับระบบที่ต้องทำ PGP encryption ในอนาคต

  • ควบคุม timing, resource, alternating order

🔄ขั้นตอนการทดสอบทีละขั้น

  • ตรวจสอบการตั้งค่า (Validate Config)
    อ่านและตรวจสอบ config.json — ถ้าค่าไหนผิดช่วง (เช่น จำนวนรอบนอกช่วง 1–1,000 ครั้ง) หยุดทันทีก่อนเริ่มทดสอบ แจ้งชื่อค่าที่มีปัญหา
  • บันทึกสภาพแวดล้อม (Detect Environment)
    บันทึกสเปก VM: vCPU (จำนวน processor), RAM (หน่วยความจำ), OS, CPU arch, storage type, สถานะ AES-NI (hardware acceleration), turbo/governor บันทึกเวอร์ชันจริงของ Go/JDK/Spring Boot/Bouncy Castle/go-crypto
  • สร้างข้อมูลและกุญแจ (Generate Shared Inputs)
    สร้างไฟล์ทดสอบแบบ deterministic (ผลเหมือนเดิมทุกครั้ง) วางบน RAM disk เพื่อตัด disk I/O ออกจากการวัด — สร้างและตรวจสอบกุญแจ RSA-2048, RSA-4096, Curve25519
  • Build ซอฟต์แวร์
    Build Go binary (ไฟล์รัน) และ Java JAR (ไฟล์รัน) บันทึกเวอร์ชัน toolchain ถ้า GraalVM native build ล้มเหลว ข้ามเฉพาะ variant นั้น ไม่หยุดทั้งชุด
  • สลับลำดับ (Alternating Execution) — หัวใจหลัก
    วนหลาย Round สลับลำดับ Go/Java ทุกรอบ: รอบ 1 = [Go, Java], รอบ 2 = [Java, Go], รอบ 3 = [Go, Java] ... เพื่อให้ทั้งสองฝ่ายได้ CPU cache และ thermal condition เท่ากัน ไม่มีฝ่ายได้เปรียบจากการรันก่อนเสมอ
  • ด่านความถูกต้อง (Verification Gate)
    ทุก operation ต้องผ่าน: ถอดรหัสกลับได้ byte-for-byte ก่อนเวลาเข้าสถิติ ตรวจ checksum (ลายนิ้วมือดิจิทัล), เวอร์ชัน, interoperability Go↔Java↔gpg
  • คำนวณสถิติ (Statistics Engine)
    คำนวณ p50/p95/p99 (ค่า percentile), mean/min/max, stddev (ค่าเบี่ยงเบน), CV, confidence interval (ช่วงความเชื่อมั่น), effect size (ขนาดผลกระทบ) คัดเลือก variant ดีที่สุดของแต่ละภาษา เปรียบเทียบ head-to-head
  • สร้างรายงาน (Report Generator)
    สร้าง results.json (ข้อมูลดิบ) และรายงาน HTML ที่อ่านได้ ต้องเสร็จภายใน 60 วินาทีหลังจบ — เขียนแบบ atomic (ครบหรือไม่มีเลย)

📏วิธีจับเวลา (Timing Methodology)

✅ สิ่งที่จับเวลา (นับเข้าผล)

  • การเรียก encrypt() function จริง — เข้ารหัส
  • การเรียก decrypt() function จริง — ถอดรหัส
  • เฉพาะ crypto transform เท่านั้น
  • Go: นาฬิกาที่แม่นยำระดับ nanosecond ไม่เดินถอยหลัง
  • Java: System.nanoTime() (เดียวกัน ความแม่นยำระดับ nanosecond)

❌ สิ่งที่ไม่จับเวลา (ไม่นับเข้าผล)

  • การโหลดกุญแจจาก disk (hard drive)
  • การอ่าน/เขียนไฟล์ (disk I/O)
  • เวลาที่ Java ใช้เปิดตัวโปรแกรม (JVM startup)
  • เวลาที่ Java อุ่นเครื่องก่อนทำงานเต็มประสิทธิภาพ
  • Warm_Up_Iteration rounds
  • Harness overhead (เวลาตัวกลาง)
Cold_Start (เวลาเปิดโปรแกรม + อุ่นเครื่องรวม) บันทึกแยกต่างหาก เป็นข้อมูลเสริม ไม่ถูกนำไปรวมกับเวลาเข้ารหัสที่ใช้เปรียบเทียบหลัก — สำคัญสำหรับ use case ที่โปรแกรมเปิด-ปิดบ่อย เช่น serverless หรือ batch ทีละไฟล์

⚖️มาตรการความยุติธรรม (Fairness Measures)

🔄 Alternating Order (สลับลำดับ)

สลับลำดับ Go/Java ทุกรอบ เพื่อลด bias (ความเอนเอียง) จากการที่ ซอฟต์แวร์หนึ่งได้ CPU warm cache (หน่วยความจำแคชที่ร้อน) ก่อนเสมอ

🔑 Input เดียวกัน (Shared Inputs)

ชุดกุญแจเดียว, ชุดไฟล์ทดสอบเดียว, crypto algorithm เดียวกัน checksum (ลายนิ้วมือดิจิทัล) ยืนยันว่า ซอฟต์แวร์ทั้งสองใช้ข้อมูล byte-for-byte เหมือนกันทุก bit

💾 Resource เท่ากัน

CPU cores (จำนวน processor) และ memory quota (โควต้าหน่วยความจำ) เท่ากันทั้งสองฝ่าย มี 1 ซอฟต์แวร์ active ต่อเวลา ไม่แย่งทรัพยากรกัน

📦 แยกโปรแกรม (แต่ละภาษารันแยกกัน)

Go และ Java รันเป็นโปรแกรมแยกกันสมบูรณ์ หน่วยความจำ, การจัดการ garbage, compiler ของ Java ไม่ส่งผลต่อ Go และในทางกลับกัน — ไม่มีการแย่งทรัพยากรกัน

🔑ชนิดกุญแจที่ทดสอบ — PGP ใช้กุญแจ 2 ชั้น

PGP ทำงานยังไง?
เหมือนการส่งของในกล่องล็อค 2 ชั้น:
ชั้นนอก (กุญแจ RSA/ECC) — ล็อคกุญแจ AES ไว้ เปิดได้เฉพาะผู้รับจริง
ชั้นใน (AES-256) — เข้ารหัสข้อมูลจริงอย่างรวดเร็วด้วย hardware

🔵 RSA-2048 — มาตรฐานทั่วไป

ใช้อยู่ในปัจจุบัน — ระบบส่วนใหญ่ใช้อันนี้

  • ขนาดกุญแจ 2,048 bit (256 ไบต์)
  • NIST รับรองถึงปี 2030
  • เร็วพอสำหรับงานทั่วไป
  • แนะนำสำหรับระบบปัจจุบัน

🟠 RSA-4096 — ความปลอดภัยสูง

ใช้เมื่อข้อมูลต้องปลอดภัยนานหลายปี

  • ขนาดกุญแจใหญ่กว่า 2 เท่า (512 ไบต์)
  • ปลอดภัยกว่า RSA-2048 มาก
  • ช้ากว่า RSA-2048 ประมาณ 3–4 เท่า
  • แนะนำสำหรับข้อมูลลับสำคัญมาก

🟣 Curve25519 ECC — ยุคใหม่

เร็วที่สุด ปลอดภัยที่สุดในบรรดาสามชนิด

  • กุญแจเล็กกว่า RSA มาก (32 ไบต์เท่านั้น)
  • ปลอดภัยเทียบเท่า RSA-3072
  • เร็วกว่า RSA อย่างชัดเจน
  • แนะนำสำหรับระบบใหม่ทุกระบบ

⚙️วิธีเขียนซอฟต์แวร์ที่ทดสอบ (Variants) คืออะไร?

ทำไมต้องมีหลาย Variant?
เหมือนการแข่งรถ — แต่ละทีมมีรถหลายคัน (สูตรต่างกัน) ก่อนเลือกคันที่เร็วที่สุดมาแข่งรอบสุดท้าย เราทดสอบแต่ละภาษาหลายแบบ แล้วเอาแบบที่เร็วที่สุดมาแข่งกัน — เพื่อให้แต่ละภาษาได้แสดงศักยภาพสูงสุด

🔵 Go มี 3 วิธี:

1. go-inmem-single
โหลดทั้งไฟล์ก่อน (1 งาน)

วิธี: โหลดไฟล์ทั้งหมดเข้า RAM ก่อน แล้วค่อยเข้ารหัสทีเดียว ทำงาน 1 ไฟล์ต่อครั้ง

✅ เหมาะกับ: ไฟล์เล็กที่ RAM พอ

⚠️ ข้อจำกัด: ไฟล์ใหญ่กิน RAM มาก

2. go-stream-single
อ่านทีละนิด (1 งาน)

วิธี: อ่านและเข้ารหัสทีละชิ้นเล็ก ๆ (64KB ต่อครั้ง) ไม่ต้องโหลดทั้งไฟล์เข้า RAM RAM ที่ใช้คงที่ไม่โตตามขนาดไฟล์

✅ เหมาะกับ: ไฟล์ขนาดใดก็ได้ RAM ประหยัด

3. go-stream-parallel
อ่านทีละนิด (หลายงานพร้อมกัน)

วิธี: เหมือน go-stream-single แต่รัน 4 งานพร้อมกัน ใช้ CPU หลาย Core พร้อมกัน เพิ่ม throughput

✅ เหมาะกับ: หลายไฟล์พร้อมกัน ต้องการ throughput สูง

🟠 Java มี 4 วิธี:

1. java-inmem-single
โหลดทั้งไฟล์ก่อน (1 งาน)

เหมือน go-inmem-single ฝั่ง Java — โหลดทั้งไฟล์เข้า RAM แล้วเข้ารหัสทีเดียว

✅ ไฟล์เล็ก

⚠️ กิน RAM ตามขนาดไฟล์

2. java-stream-single
อ่านทีละนิด (1 งาน)

เหมือน go-stream-single ฝั่ง Java — อ่านทีละ 64KB ประหยัด RAM

✅ ไฟล์ขนาดใดก็ได้

3. java-stream-parallel
อ่านทีละนิด (หลายงาน)

อ่านทีละ 64KB + ทำงาน 4 งานพร้อมกัน ใช้ CPU หลาย Core

✅ throughput สูงสุดของ Java

4. java-native-stream-parallel
compile เป็น native binary

Compile Java ล่วงหน้าเป็น binary เหมือน Go — ไม่ต้องรอ JVM เปิด เริ่มเร็วกว่าปกติมาก

ℹ️ ยังไม่ได้ทดสอบรอบนี้ (ต้องติดตั้ง GraalVM เพิ่ม)

🏆 เกณฑ์คัดเลือก "ตัวแทน" ของแต่ละภาษา: เลือก variant ที่มีเวลาเข้า+ถอดรหัสรวม (round-trip) น้อยที่สุด ถ้าเท่ากัน เลือกตัวที่ใช้ RAM น้อยกว่า variant ที่ให้ผลผิดพลาด (ข้อมูลหลังถอดรหัสไม่ตรง) ถูกตัดออกก่อนเลย

🔒สูตรการเข้ารหัสที่ใช้ (Crypto Profile)

ค่าคงที่เดียวกันทุก variant ทุกภาษา — เพื่อให้เปรียบเทียบได้อย่างยุติธรรม เหมือนการแข่งขันที่ทุกคนใช้สนามเดียวกัน กติกาเดียวกัน

🔐 AES-256 — การเข้ารหัสข้อมูลจริง

AES คือ "ตัวล็อค" ที่เข้ารหัสข้อมูลจริง ใช้กุญแจขนาด 256 bit เป็นมาตรฐานสากลที่รัฐบาลสหรัฐฯ และองค์กรทั่วโลกใช้

ทำไมถึงเร็ว? CPU สมัยใหม่มีวงจรพิเศษ (AES-NI) ที่ทำ AES ได้โดยตรง เหมือนคิดเลขด้วยเครื่องคิดเลขแทนการคิดในหัว — เร็วกว่ากันมาก VM ของเราเปิดฟีเจอร์นี้ไว้แล้ว

🗜 ZLIB — การบีบอัดก่อนเข้ารหัส

ก่อนเข้ารหัส ระบบบีบไฟล์ให้เล็กลงก่อน ทำให้ AES ทำงานน้อยลง = เร็วขึ้น

ไฟล์บีบได้ (.txt, .csv): ขนาดลดลง 60–80% ก่อนเข้ารหัส ประหยัดทั้งเวลาและพื้นที่
ไฟล์บีบไม่ได้ (.pdf ที่มีรูป, .zip): ZLIB ลองบีบแล้วไม่ได้ผล ข้ามไปเลย ไม่ทำให้ช้าลงมาก

ชั้นที่ทำอะไรเทคนิคที่ใช้ความสำคัญ
1. กุญแจสาธารณะ ล็อคกุญแจ AES ให้เฉพาะผู้รับเปิดได้ RSA-2048 / RSA-4096 / Curve25519 ความปลอดภัย — เฉพาะผู้รับอ่านได้
2. บีบอัดข้อมูล ย่อไฟล์ให้เล็กก่อนเข้ารหัส ZLIB ประหยัดเวลา + พื้นที่จัดเก็บ
3. เข้ารหัสข้อมูลจริง ล็อคเนื้อหาไฟล์ด้วย AES AES-256 (เร็วด้วย CPU hardware) งานหลัก — เร็วที่สุดในทุกชั้น
4. ลายนิ้วมือ ตรวจว่าข้อมูลไม่ถูกแก้ไขระหว่างทาง SHA-256 ตรวจจับการปลอมแปลง

📋Property-Based Testing (PBT) คืออะไร? — อธิบายให้ไม่รู้ซอฟต์แวร์เข้าใจ

🔵 Unit Test ทั่วไป — แบบที่คุ้นเคย

เหมือนการทดสอบสูตรอาหารด้วยตัวอย่างที่เลือกเอง:
"เอา แป้ง 100g + ไข่ 1 ฟอง → ควรได้เค้ก 1 ชิ้น"
ครอบคลุมเฉพาะ case ที่นักพัฒนานึกออก ถ้าลืม edge case บางอย่าง (เช่น แป้ง 0g?) — test จะไม่ตรวจ

ข้อจำกัด: ถ้าไม่นึกถึง input แปลก ๆ test จะไม่เห็น bug

🟢 Property-Based Test — แบบที่ใช้ใน POC นี้

เหมือนการกำหนด "กฎธรรมชาติ" แล้วให้ระบบสุ่มทดสอบเอง:
"ถ้าเข้ารหัสข้อมูลอะไรก็ตาม แล้วถอดรหัส ต้องได้ข้อมูลเดิมเสมอ"
Framework สุ่ม input 200 แบบอัตโนมัติ — รวมถึงข้อมูลว่าง, binary แปลก ๆ, ขนาดสุดโต่ง — ที่นักพัฒนาอาจไม่นึกถึง

ข้อดี: ค้นหา bug ที่ซ่อนอยู่ใน edge case ที่ไม่มีใครคาดคิด

ตัวอย่างกฎ (Properties) ที่ทดสอบ:
① ถอดรหัส(เข้ารหัส(X)) = X เสมอ — ไม่ว่า X จะเป็นอะไร (ข้อมูลว่าง, 50 KB, binary, ฯลฯ)
② หน่วยความจำสูงสุดระหว่าง streaming ต้องไม่โตตามขนาดไฟล์ (คงที่ที่ <12 MB)
③ เวลาเข้ารหัส + เวลาถอดรหัส ≤ เวลารวมที่รายงาน เสมอ
Framework ที่ใช้: jqwik (Java), pgregory.net/rapid (Go), hypothesis (Python) — รัน ≥100 iterations ต่อ property

🔵Go Tests — รายละเอียด

Property 1: Round-trip byte-for-byte

กฎ: ถอดรหัส(เข้ารหัส(X)) = X ทุก byte สำหรับ payload ทุกรูปแบบ:

  • Empty file (0 bytes — ไฟล์ว่าง)
  • Lone CR ( ) — ต้องไม่ถูก normalize เป็น CRLF
  • ทุก 256 byte values (0x00–0xFF)
  • Compressible text (.txt, .csv — ข้อความบีบอัดได้)
  • Incompressible binary (.pdf-like — binary บีบไม่ได้)
  • Medium payloads 64 KB–512 KB

Property 3: File classification rules

💡JVM Warm คืออะไรกันแน่? — อธิบายเพิ่มเติมสำหรับคนทั่วไป

คำว่า "JVM อุ่น" (JVM Warm) มักเข้าใจผิดว่าหมายถึง "ข้อมูลค้างในแรม" ซึ่งไม่ถูกต้องทั้งหมด ขอแยกอธิบาย 2 แนวคิดที่ต่างกัน:

🔥 JVM JIT Warm (สิ่งที่เกิดขึ้นในรอบแรก)

JIT = Just-In-Time Compiler — ตัวแปลโค้ดแบบ real-time

JVM ไม่ได้รันโค้ดตรงๆ แต่แปลงเป็น "native instructions" ขณะรัน รอบแรกๆ JIT ยังไม่ได้แปลงฟังก์ชัน crypto — รันแบบ "interpreted" (ช้า) พอรันซ้ำหลายครั้ง JIT เรียนรู้และ compile ฟังก์ชันนั้นให้เร็วขึ้น

ผลที่เกิด: JVM ที่ "อุ่นแล้ว" (ผ่านการรันหลายรอบ) จะเร็วกว่า JVM ที่เพิ่งเริ่ม แม้ ข้อมูลที่รันจะไม่เหมือนกันเลย เพราะ JIT optimize โค้ดฟังก์ชัน ไม่ใช่ข้อมูล

ตอบคำถาม "ถ้าข้อมูลไม่ซ้ำกัน แรมอุ่นช่วยได้มั้ย?"
✅ ช่วยได้ — เพราะ JVM Warm ไม่ได้ cache ข้อมูล แต่ cache โค้ด crypto ที่ compiled แล้ว ข้อมูลใหม่ทุกไฟล์ก็ยังได้ประโยชน์จาก JIT ที่อุ่นแล้วเต็มที่

💾 RAM/OS Cache (สิ่งที่ไม่เกี่ยวกับ JIT)

OS cache ไฟล์ที่เพิ่งอ่านไว้ใน RAM — ถ้าอ่านไฟล์เดิมอีกครั้ง เร็วกว่าอ่านจาก disk

ในการทดสอบนี้: ไฟล์ทดสอบวางบน tmpfs (RAM disk) อยู่แล้ว — ดังนั้น OS file cache ไม่มีผลต่อผลการทดสอบ

สำหรับชีวิตจริง: ถ้าข้อมูลไม่ซ้ำกัน (เช่น user อัพโหลดไฟล์ใหม่ทุกครั้ง) OS file cache ก็ไม่ช่วยอยู่แล้ว — ข้อมูลแต่ละชิ้นต้องอ่านใหม่เสมอ

ดังนั้น RAM cache ของข้อมูล ไม่เกี่ยวกับ JVM Warm สองอย่างนี้แยกกันสมบูรณ์

🧪 ผลทดสอบ Warm JVM บน VM จริง — RSA-2048 · .txt · 15 files × 512KB

ทดสอบเพิ่มเติมบน VM เดิม (Ubuntu 24.04, 8 vCPU, 14GB RAM) ยังไม่ reboot เพื่อ confirm ว่า JIT warm-up ช่วย Java จริงแค่ไหน

Latency Statistics (สถิติเวลาตอบสนอง)

  • p50 (median — ค่ากลาง): เรียงทุกค่าแล้วเอาตัวกลาง — 50% ของ operations เร็วกว่านี้
  • p95: 95% ของ operations เร็วกว่านี้ (reliable เมื่อ n≥20 ค่า)
  • p99: worst-case ที่ยอมรับได้ (reliable เมื่อ n≥100 ค่า)
  • min/max/mean: ต่ำสุด/สูงสุด/เฉลี่ย
  • stddev (ค่าเบี่ยงเบนมาตรฐาน): บอกว่าตัวเลขกระจายตัวมากแค่ไหน
  • CV = stddev/mean: ความผันผวนสัมพัทธ์
  • Method: linear interpolation type-7

Throughput & Additional (ปริมาณงานและอื่น ๆ)

  • MB/sec = bytes/1,048,576 ÷ time_s (crypto-only)
  • files/sec = files ÷ time_s
  • Round-trip ms = encryptMs + decryptMs
  • Error rate = failures/attempted ∈ [0.0, 1.0]
  • Confidence interval 95% (ช่วงที่เชื่อมั่น 95% ว่าค่าจริงอยู่ในนั้น)
  • Effect size (Cohen's d) (ขนาดผลกระทบ — small/medium/large)
  • Inconclusive ถ้า diff ≤5% (ต่างกันน้อยจนไม่มีนัยสำคัญ)

⚠️ข้อจำกัดและหมายเหตุสำคัญ

สิ่งที่ต้องระวังในการตีความ

  • VM ≠ Bare Metal: ผลบน VM อาจต่างจาก bare metal 10–20% แต่ความสัมพันธ์ relative Go vs Java น่าจะยังคงเดิม
  • JVM Startup ไม่รวมใน core metric: Java ใช้เวลา ~1.4 s/round JVM warmup แต่ไม่ถูกนับ สำหรับ long-running service ผลต่างนี้ไม่มีนัยสำคัญ
  • GraalVM Native ยังไม่ได้ทดสอบ: native-image ไม่ได้ติดตั้งบน VM — อาจทำให้ Java startup ดีขึ้นมาก
  • เฉพาะ AES-256+ZLIB+SHA-256: cipher/compression อื่น (เช่น ChaCha20) อาจให้ผลต่าง

Reproducibility (ทำซ้ำได้)

  • Corpus สร้างจาก seed เดิมทุกครั้ง (bit-for-bit identical)
  • Key_Set checksum ยืนยันก่อนทุก run
  • เวอร์ชัน toolchain บันทึกใน results.json
  • config.json เก็บค่าที่ใช้จริงทั้งหมด
  • Raw per-operation samples เก็บครบทุกค่า
  • ทำซ้ำได้บน VM เดิมโดยรัน benchmark script ใหม่

➡️Next Steps — ข้อเสนอแนะสำหรับผู้บริหาร

จากผลการทดสอบนี้ ขั้นตอนต่อไปขึ้นอยู่กับบริบทของโครงการ:

📌 ถ้าเลือก Go (แนะนำสำหรับระบบใหม่)

  • เหมาะที่สุดสำหรับระบบที่ต้องการ PGP throughput (ปริมาณงาน) สูงสุด
  • ใช้ Curve25519 ECC เป็น key type หลัก — เร็วกว่า RSA 3.6–3.9×
  • ทดสอบ workload จริงบน bare metal (เครื่องจริงไม่ใช่ VM) ก่อน go-live
  • ทีมต้องมีความรู้ Go — ควรประเมิน learning curve
  • Library ProtonMail/go-crypto ดูแลต่อเนื่อง มี community ดี

📌 ถ้าใช้ Java ต่อ (ระบบที่มีอยู่แล้ว)

  • Java ทำได้ดีพอ แค่ช้ากว่า 1.4–3.9× ขึ้นอยู่กับ key type
  • ลองทดสอบ GraalVM Native Image — อาจลด startup time ได้มาก
  • ใช้ Curve25519 แทน RSA ถ้าเป็นระบบใหม่ — ช่วยความเร็วได้มาก
  • ใช้ streaming variant แทน in-memory — ลด peak memory ลง
  • Bouncy Castle library ดูแลต่อเนื่อง มี enterprise support

🔬 ข้อแนะนำสำหรับ POC รอบถัดไป (ถ้าต้องการข้อมูลเพิ่ม)

  • ทดสอบบน bare metal (เครื่องจริง) — ผลแม่นกว่า VM 10–20%
  • ติดตั้ง GraalVM Native Image ทดสอบ java-native variant ให้ครบ
  • เพิ่ม rounds เป็น ≥30 — สถิติ p95/p99 มีความน่าเชื่อถือสูงขึ้น
  • ทดสอบ concurrent load (หลาย user พร้อมกัน) ไม่ใช่แค่ single-run
  • ทดสอบ memory pressure — เมื่อ RAM ไม่พอ streaming variant ได้เปรียบชัดเจน
  • ทดสอบ large files >100 MB — ผลอาจต่างจากไฟล์ 5 MB
  • ทดสอบ signing + verification ถ้า use case ต้องการ
  • ทดสอบ ChaCha20 เป็น cipher อีก profile
